# 目标

1. 将项目中的测试体系从 JUnit4 迁移到 JUnit5，并确保 Maven 默认测试流程可以正常执行。
2. 将原先基于 JUnit4 Runner 的 `ApplicaiontContextRunner` 改造为 JUnit5 下更合适的扩展机制，继续提供“测试实例由 `ApplicationContext` 创建”的能力。

# 核心思路

1. 先调整 `pom.xml`，移除 JUnit4 依赖，补齐 JUnit5 所需的 API、引擎与 Suite 支持，并显式配置 Surefire 运行 Jupiter 测试。
2. 保留 `cc.jfire.jfire.helpjunit.ApplicaiontContextRunner` 这个入口类名，避免无谓的 API 断裂，但将其实现从 `BlockJUnit4ClassRunner` 改为 JUnit5 Extension。
3. 为了让扩展的使用方式更符合 JUnit5 习惯，增加一个面向测试类的组合注解，统一挂载扩展；扩展内部负责创建 `DefaultApplicationContext`，并从容器中获取测试类实例。
4. 批量迁移现有测试代码：将 `org.junit.*` 的测试注解、断言和 Suite 入口切换到 JUnit5 等价写法，保证现有测试语义不变。
5. 增加或调整针对 `ApplicaiontContextRunner` 的测试，覆盖 JUnit5 扩展模式下的核心能力，避免只完成依赖迁移而没有验证新测试辅助机制。

# 详细思路

## 1. 测试依赖与构建链路迁移

当前 `pom.xml` 仍依赖 `junit:junit:4.12`，而项目测试代码全部使用 JUnit4 的 `@Test`、`Assert`、`@RunWith` 和 `Suite`。迁移时需要同时处理“编译依赖”和“测试执行器”两部分：

- 主代码中的 `ApplicaiontContextRunner` 会直接引用测试框架 API，因此需要为主代码保留 JUnit5 API 依赖。
- 项目测试运行需要引入 Jupiter Engine。
- `Coverage` 这种聚合入口需要改成 JUnit Platform Suite，因此还需要引入对应 suite 依赖。
- 为避免 Maven 在不同版本默认行为下无法发现 Jupiter 测试，需要在 `maven-surefire-plugin` 中显式启用合适版本。

## 2. 将 Runner 模式改为 Jupiter Extension

`ApplicaiontContextRunner` 当前继承 `BlockJUnit4ClassRunner`，其核心价值不是 JUnit4 本身，而是“测试实例不是直接反射创建，而是通过 `DefaultApplicationContext` 获取 Bean”。在 JUnit5 里，更合适的实现方式是 Extension：

- 扩展负责按测试类创建并缓存 `ApplicationContext`。
- 扩展负责生成测试类实例，让测试对象仍然由容器托管。
- 若有需要，可顺带提供 `ApplicationContext` 参数解析能力，方便测试方法直接声明上下文参数。

这样既能保留原辅助类的语义，又符合 JUnit5 的扩展模型，而不需要继续模拟 Runner 的执行链。

## 3. 提供更自然的测试接入方式

JUnit5 不再推荐通过 `@RunWith` 控制整个测试类生命周期，因此应新增一个面向 Jfire 测试的组合注解，例如 `@JfireJUnitSupport` 或同类命名：

- 该注解内部使用 `@ExtendWith(ApplicaiontContextRunner.class)` 挂载扩展。
- 测试类只需添加一个注解即可启用容器化测试实例创建。
- 这样比直接暴露扩展细节更利于后续演进，也便于项目 README 或示例代码统一描述用法。

## 4. 批量迁移测试代码

当前测试代码中共有多处 JUnit4 直接引用，迁移需要保证行为保持一致：

- `org.junit.Test` 改为 `org.junit.jupiter.api.Test`
- `org.junit.Assert` 与 `org.junit.Assert.*` 改为 `org.junit.jupiter.api.Assertions`
- `Coverage` 从 `@RunWith(Suite.class)` 与 `@Suite.SuiteClasses` 改为 JUnit Platform Suite 注解

现有测试大多手动创建 `DefaultApplicationContext`，因此测试逻辑本身不需要重写，只要更新 API 即可。真正需要体现新扩展能力的，是专门围绕 `ApplicaiontContextRunner` 的辅助测试。

## 5. 验证重点

开发时应重点确认以下几类结果：

1. Maven 执行测试时可以发现并运行 Jupiter 测试。
2. 原有测试在迁移后仍保持相同断言语义。
3. `ApplicaiontContextRunner` 在 JUnit5 下能正确从 `DefaultApplicationContext` 获取测试类实例。
4. `Coverage` 作为聚合测试入口在 JUnit5 下仍可正常工作。
